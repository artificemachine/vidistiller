1. **Alembic is decorative — schema is managed exclusively by `main.py`'s startup lifespan, on every process boot.** `backend/app/main.py:107-126` runs `Base.metadata.create_all()` plus 3 hand-written `ALTER TABLE ADD COLUMN` statements (each independently try/rollback-wrapped). `migrations/versions/0001_squashed_baseline.py:31-32` — the *sole* Alembic revision, created 2026-07-21 — has an `upgrade()` that itself just calls `Base.metadata.create_all()` again; its own docstring admits "the previous migration chain was unrunnable... alembic had drifted into a broken, decorative state." Nothing anywhere invokes `alembic upgrade head` (not in `main.py`, any deploy script, or any compose file); `scripts/setup-staging.sh:92` explicitly comments `# Base metadata build handles schema if alembic not wired`. **Real consequence**: `create_all()` only creates missing *tables*, never alters existing ones — any future column added to a model without a matching manual entry in the `main.py:116-119` ALTER-loop will silently be missing on any pre-existing database (staging/prod), surfacing only as a runtime `column does not exist` error on first use. No test protects against this drift.
2. **The squash destroyed rollback capability.** `0001_squashed_baseline.py:35-36`'s `downgrade()` does `Base.metadata.drop_all()` — a full destructive wipe presented as a migration downgrade, not a real rollback path.
3. **Startup-time DDL under concurrent boot.** The `create_all` + 3 ALTERs run inside `lifespan` on every process start; with multiple Uvicorn/Celery workers restarting concurrently (a rolling deploy), concurrent `ALTER TABLE ADD COLUMN` from separate processes can race on Postgres DDL locks — currently only "mitigated" by the try/rollback swallowing the resulting error, not by any real coordination.

Not re-litigated: FK/index hygiene (`ondelete=`, indexes, unique constraints) inspected and looks reasonable — not the systemic gap here.
